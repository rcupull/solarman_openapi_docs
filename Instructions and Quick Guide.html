Instructions and Quick Guide

View on Yuque
1 Open platform description
1The SolarMAN OpenAPI open platform provides standardized RESTful data calling services for authenticated users.
2The platform interface is based on the http/https protocol and uses the OAuth2.0 authentication mechanism.
3Use GET/POST, see the interface definition for details. The parameters are utf-8 encoded and urlencode processed.
4The data is encoded using utf-8
5Supports returning json format
6Return data structure: body content, code: error code, msg: error content
7In addition to obtaining and extending the ACCESS_TOKEN interface, when calling other interfaces, you need to add Authorization in the http header, and its Value is the value of ACCESS_TOKEN.
1.1 Platform terminology
application
The basic elements of the open platform. Users can call the platform API through created applications.
APP_ID
The unique identification ID of the application
APP_SECRET
The authentication key of the application needs to be kept properly and cannot be leaked.
ACCESS_TOKEN
The authorization token after successful authentication is used to request server data.
REFRESH_TOKEN
Update token, used to extend the validity period of ACCESS_TOKEN

1.2 Access process
1Register as a user of Wheat Smart Electronics or Wheat Merchant Edition
When applying for API login elements, please provide
[User contact information]: mainly email, and other content that can assist in identity authentication (such as solarman account)
[Customer type]: individuals, equipment vendors, installation and operation and maintenance vendors, distributors, investors
[Description of purpose of use]: Clarify the usage scenario

Send the email to your business contact person or after-sales email:service@solarmanpv.com

2Register as a SOLARMAN developer (contact business), create an application APP, generate APP_ID and APP_SECRET, and submit for business review
3Developers conduct business development and testing based on development documents
4Formal environment docking interface

1.5 Quick Guide
1.5.1 Processing data from the "Wheat Intelligence" platform through OpenAPI
1.5.1.1 Obtain power plant and equipment data

Prerequisites (just meet any one)
(1) The account has created a power station and added a collector in the Wheat Smart Web/App or OpenAPI, and the data can be viewed normally;
(2) The account has received the power station authorized by the merchant on the Wheat Smart Web/App, and the data can be viewed normally.
Call guide
The calling guide only demonstrates some of the scenarios marked in the flow chart. If you need to know the usage of other interfaces, please refer to the specific documentation below.
Step 1: Get Token
Before using OpenAPI, you need to exchange your developer information and "Wheat Smart" platform account for the OpenAPI calling Token. The Token will be the only certificate and necessary condition for you to call all other interfaces in OpenAPI.
(1) Interface coding:2.1
(2) Things to note:
●The APPID parameter is a query parameter that needs to be placed on the URL.
●The password field needs to be encrypted with SHA256 before passing parameters.
●Calling multiple times to obtain the token will not cause the original token to become invalid. The token is valid for two months and will become invalid upon expiration. When resetting the password or modifying the user role, the original token will also become invalid.
●The APPID of the test environment cannot call the production environment API.
●You must have an account on the "Wheat Smart Electronics" platform to obtain the Token normally. Please enter one of the mobile/email/username parameters based on the actual binding of the account, and bring your password.
Step 2: Get the list of power stations under the account
Through this interface, you can obtain the list of power stations under the current account and learn the Plant ID of each power station.
(1) Interface coding:4.4
(2) Things to note:
●The header field authorization must be passed. When passing the value, you need to spell the token_type prefix "bearer" (please note that there is a space after bearer)
○Example:
■Obtained TOKEN value: FFFFFFFFF
■The final authorization field value of Header is:
●When pagination parameters are not passed, the first page is queried by default.
Step 3: Obtain real-time data of the power station
After knowing the Plant ID of the power station, you can query the real-time data of the power station through this interface, such as: power station power generation, power station grid-connected power, etc.
(1) Interface coding:4.5
(2) Things to note:
●The stationId must be passed. Only the station ID that matches the user can be passed.
Step 4: Obtain the list of equipment under the power station
After knowing the Plant ID of the power station, you can query the device list under the power station through this interface and learn the SN and Device ID of each device.
(1) Interface coding:4.2
(2) Things to note:
●If the field deviceType is not passed, all types of device data will be queried by default. If a value is passed, the specified type of device will be queried.
●Please refer to the API documentation for device types.7. Enumeration type description
●When pagination parameters are not passed, the first page is queried by default.
Step 5: Obtain real-time data of the device
After knowing the device SN and Device ID, you can query the real-time data of the device through this interface, such as: total AC output power of the inverter, component power generation, battery charging power, etc.
(1) Interface coding:3.3
(2) Things to note:
●The field deviceSn must be passed
